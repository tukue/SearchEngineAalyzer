# PR Relevance Review (DevOps)

## Context
- The Express variant serves API routes and the built frontend on port 5000 in both development and production (`server/index.ts`).
- The Express build uses a single `npm run build` step, bundling the server with esbuild and emitting static assets to `dist/public` (`package.json`, `vite.config.ts`).
- CI quality gates run `bash run-tests.sh`, which builds the full stack, starts the production server, and exercises API + UI checks before deployment.
- The container image for the Express variant follows the same pattern: install dependencies, run `npm run build`, then start the production server from the bundled output at runtime (`Dockerfile`).
- A Next.js variant also exists; it expects its own build pipeline (`next build`), routing layer, and (optionally) Vercel-native data-fetching/runtime features. The deployment strategy must match the chosen variant.

## PR under review
The current deployment workflow added in the PR deploys only the static frontend output to Vercel using the v3 prebuilt format (`.github/workflows/deploy.yml`). It intentionally skips uploading any server code and relies on Vercel as a static host.

## Relevance to the current application state
- **Server-backed features are required for both variants.** For the Express build, the server handles API routes and serves the client from the same process (`server/index.ts`), so a purely static Vercel deploy will not provide the backend that `run-tests.sh` and the Dockerfile expect. For the Next.js build, API routes, middleware, and server actions also require serverful hosting; a static-only export would break those paths.
- **Build artifacts diverge by variant.** The workflow currently builds only the Express frontend (`npm run build:frontend`) before staging files under `.vercel/output`. The Express runtime build (`npm run build`) bundles the server entrypoint to `dist/index.js`. The Next.js runtime build would require `next build` and the `.vercel/output` format generated by `next export` or the default Next.js build. Skipping the appropriate backend build for either variant means no backend artifact is produced for deployment.
- **Tests assume a full-stack environment.** The CI step runs `bash run-tests.sh`, which starts the Express production server (`npm run start`) to serve both API and UI during checks. A Next.js pipeline would need equivalent smoke tests (e.g., `next build && next start` or `vercel dev`) to ensure parity. Deploying only static assets means those checks are not representative of what actually runs on Vercel.
- **Container and deployment targets differ.** The Dockerfile and start scripts run the full Express server on port 5000, but the Vercel workflow publishes a static site with an SPA route rewrite. If the Next.js variant is the target, its Docker/runtime shape should match the Next.js server (e.g., `next start` or Vercel functions). Any operational runbooks or smoke tests built around the Express container will not cover the Vercel-hosted version.

## Recommendations
- Decide and document which variant is being deployed to Vercel (Express or Next.js) and keep the workflow + runtime consistent with that choice.
- If Vercel should host only a frontend, document and provision a companion backend (e.g., fly.io/Render/Docker host) and configure the frontend to call it via environment-specific base URLs.
- If Vercel is expected to run the full app, switch to a serverful deployment (e.g., Vercel Functions or another PaaS) that builds and ships the correct backend output for the chosen variant alongside the static assets.
- Align CI and release workflows so the artifacts and runtime topology (full stack vs. static SPA) match, ensuring post-deploy smoke tests target the same environment shape.

### Path to run the full stack on Vercel (Express variant)
- **Refactor the server to export a handler** rather than immediately calling `listen` so it can be required by a function entrypoint (e.g., expose an `export const app` from `server/index.ts`).
- **Add a Vercel function wrapper** (such as `api/server.ts`) that imports the Express app, wraps it with `serverless-http`, and exports the handler expected by Vercel Functions. Ensure the build step emits this file into `.vercel/output/functions/api/server.func/` via the build output API or `vercel.json`'s `functions` mapping.
- **Keep the unified build command** by continuing to run `npm run build`, which outputs `dist/index.js` and `dist/public`; configure `vercel.json` to use `dist/public` for static assets and point rewrites to the function handler for `/api` and SPA fallbacks.
- **Align CI smoke tests** by adding a job that builds with the Vercel build command and runs the function handler locally via `vercel dev` or `@vercel/node` to ensure parity with the deployed shape.

### Path to run the full stack on Vercel (Next.js variant)
- **Use the native Next.js build** (`next build`), ensuring any API routes or server actions remain enabled. If opting into the Build Output API, let the Next.js build emit `.vercel/output` artifacts without downgrading to a static export unless the app is truly static.
- **Configure rewrites and edge/server functions** via `vercel.json` or the Next.js routing layer so API routes resolve to the correct serverless/edge handlers. Avoid SPA-only rewrites that would shadow Next.js API routes.
- **Adopt Next.js-aware CI smoke tests** that run `next build` and `next start` (or `vercel dev`) to verify server routes, middleware, and ISR/SSG paths before deployment.
- **Align Docker/runtime shape** if containerizing the Next.js variant (e.g., `next start` in production) so local ops workflows mirror the Vercel deployment shape.
